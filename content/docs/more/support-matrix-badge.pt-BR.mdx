# Badge da matriz de suporte

O `napi.rs` serve a matriz de suporte multiplataforma do seu pacote como uma
única imagem autocontida, gerada inteiramente a partir da query string da URL.
Aponte um `<img>` ou `<picture>` do README para ela e o badge renderiza os chips
de plataforma, as cores de cada tier e a linha de versões do Node.js — sem etapa
de build, sem asset para gerar, sem arquivo para versionar. A imagem é uma função
pura da sua query, então a edge a cacheia com folga e ela só muda quando você
muda a URL.

## Dois endpoints

| Endpoint | Use para |
| --- | --- |
| `GET /support-matrix.svg?<query>` | READMEs do GitHub. Combine dois em um `<picture>` para alternar claro/escuro. |
| `GET /support-matrix.png?<query>` | npm e crates.io, cujos READMEs removem `<picture>` e SVG inline. |

Ambos aceitam a mesma query e usam o tema claro por padrão.

## Monte sua URL

Você raramente precisa escrever a query à mão. O construtor interativo em
[/support-matrix](/support-matrix) permite escolher os triples por
tier, informar seu intervalo de `engines` e copia os trechos prontos de
`<picture>` e PNG para você. O restante desta página documenta o contrato que o
construtor emite.

## Parâmetros de query

| Parâmetro | O que controla |
| --- | --- |
| `tested` | Rust target triples que o CI executa e usa como gate, separados por vírgula — chips verdes. O literal `full` expande para o conjunto do scaffold do `napi new`. |
| `nonblocking` | Triples cujo job de CI é não-bloqueante (`continue-on-error`) — chips âmbar. |
| `untested` | Triples que são compilados mas não exercitados no CI — chips cinza. |
| `omit` | Triples ou nomes de grupo de SO a remover depois que os tiers resolvem. Grupos: `macos`, `windows`, `linux`, `android`, `freebsd`, `openharmony`, `browser`. |
| `engines` | O intervalo `engines.node` do pacote, ex.: `^22.20 \|\| ^24.12 \|\| >=25`. Deriva o título `vMIN → vMAX` e a lista de majors excluídas e intervalos parciais. |
| `nodeTested` | Majors do Node que o CI executa, separadas por vírgula, ex.: `22,24`. Marca essas pílulas de versão como testadas. |
| `wasm` | `1` força o card Browser (WASI). Ele também aparece automaticamente quando há um triple `wasm32-wasi*`. |
| `name` | Nome cosmético do pacote exibido no título e no texto alt da imagem. |
| `theme` | `light` (padrão) ou `dark`. |

Os valores são Rust target triples canônicos (`x86_64-unknown-linux-gnu`,
`aarch64-apple-darwin`, …). As duas grafias alternativas comuns
`wasm32-wasi-preview1-threads` e `arm-linux-androideabi` são aceitas e
normalizadas para sua forma canônica.

## Como a query resolve

O serviço aplica uma ordem fixa e nunca lança erro: triples desconhecidos, nomes
de grupo desconhecidos e parâmetros malformados são ignorados, então uma URL
levemente errada ainda renderiza algo sensato.

1. **`full` semeia um tier** com o conjunto do scaffold — exatamente os targets
   que o `napi new` gera.
2. **Triples explícitos sobrepõem a semente.** Nomear um triple em qualquer tier
   vence o tier que o `full` tinha semeado para ele.
3. **O tier mais severo vence** quando o mesmo triple aparece em dois tiers:
   `untested` > `nonblocking` > `tested`.
4. **`omit` subtrai por último**, por triple ou por grupo de SO.
5. **Plataformas que você nunca compila simplesmente ficam de fora** — não existe
   tier "sem suporte", então deixe esses triples fora de todas as listas.

## Como incorporar

Para um README do GitHub, sirva um SVG claro e um escuro a partir de um único
`<picture>` para que o badge acompanhe o tema do leitor:

```html
<picture>
  <source
    media="(prefers-color-scheme: dark)"
    srcset="https://napi.rs/support-matrix.svg?theme=dark&tested=full&engines=%5E22.20+%7C%7C+%5E24.12+%7C%7C+%3E%3D25"
  />
  <img
    alt="napi-rs support matrix"
    src="https://napi.rs/support-matrix.svg?theme=light&tested=full&engines=%5E22.20+%7C%7C+%5E24.12+%7C%7C+%3E%3D25"
  />
</picture>
```

npm e crates.io removem `<picture>`, então use um único PNG neles:

```md
![support matrix](https://napi.rs/support-matrix.png?tested=full&engines=%5E22.20+%7C%7C+%5E24.12+%7C%7C+%3E%3D25)
```

## Exemplo completo

O badge do `@napi-rs/lzma` marca dois triples Linux como não-bloqueantes, mais
quatro como não testados, acompanha Node 22 e 24 no CI e suporta Node
`^22.20 || ^24.12 || >=25`:

```
https://napi.rs/support-matrix.svg?tested=full
  &nonblocking=powerpc64le-unknown-linux-gnu,s390x-unknown-linux-gnu
  &untested=riscv64gc-unknown-linux-gnu,aarch64-linux-android,arm-linux-androideabi,wasm32-wasi-preview1-threads
  &engines=%5E22.20+%7C%7C+%5E24.12+%7C%7C+%3E%3D25&nodeTested=22,24&name=%40napi-rs%2Flzma
```

Isso renderiza um título de Node `v22.20 → v26` com `23` e `24.0–24.11` marcados
como excluídos, os chips ppc64le e s390x em âmbar, e os chips riscv64, ambos os
Android e o Browser em cinza. As quebras de linha acima são apenas para leitura —
envie tudo em uma única linha (o construtor faz o percent-encoding dos espaços em
`engines` para você).
