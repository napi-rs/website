# Perguntas Frequentes

## Compilar para `Linux alpine`

> https://github.com/rust-lang/rust/pull/40113#issuecomment-323193341

Você não pode definir o `crate-type` de compilação para `cdylib` ao compilar para o destino `*-unknown-linux-musl` por padrão.

Se você deseja fazer isso, precisa passar `-C target-feature=-crt-static` para `rustc`.

A CLI do **NAPI-RS** cuida disso para você: para qualquer target `*musl*`, o `napi build` adiciona automaticamente `-C target-feature=-crt-static` à variável de ambiente `RUSTFLAGS`, e o projeto gerado pelo `napi new` já compila seus targets musl dessa forma por padrão.

Como a CLI exporta isso via a variável de ambiente `RUSTFLAGS`, quaisquer `rustflags` no seu `.cargo/config.toml` são ignorados para builds musl (variáveis de ambiente têm precedência no Cargo). Se você precisa de flags extras do `rustc` para um target musl, adicione-as à variável de ambiente `RUSTFLAGS` em vez do `.cargo/config.toml`.

Se você usa o alocador [`mimalloc`](https://github.com/purpleprotocol/mimalloc_rust), habilite a feature `local_dynamic_tls` dele para targets musl; caso contrário, o addon pode falhar em tempo de execução com um erro de alocação de thread-local storage.

Veja [Compilação cruzada](../cross-build) para saber como compilar targets musl a partir de qualquer host.

## `GLIBC_x.yy` not found

```
Error: /lib/x86_64-linux-gnu/libc.so.6: version `GLIBC_2.38' not found
```

Um addon `*-linux-gnu` linka a glibc dinamicamente e exige pelo menos a versão de glibc da máquina que o **compilou**. Se o host de build roda uma distro mais nova que o host de deploy, o addon falha ao carregar com esse erro.

A correção é compilar contra uma glibc mais antiga:

- Em um host Linux x64/arm64, compile com `--use-napi-cross` — ele fixa o piso de glibc em **2.17**, que carrega em praticamente qualquer distro glibc.
- A partir de um host macOS ou Windows, compile com `--cross-compile` (`-x`) — o piso passa a ser a glibc padrão do zig: mais nova que 2.17, mas independente da distro do seu host.

**Não** mude para um target `*-musl` para consertar esse erro — musl é uma libc diferente, para distros no estilo Alpine, e não uma forma de reduzir um requisito de glibc. Veja [Versões de glibc](../cross-build#vers%C3%B5es-de-glibc) para os detalhes.

## rustls / `aws-lc-sys` falha com `--use-napi-cross` no aarch64

Se o seu crate depende de `rustls` — muitas vezes transitivamente, via `reqwest` ou `hyper-rustls` — o backend padrão dele, `aws-lc-sys`, falha ao compilar de forma cruzada para `aarch64-unknown-linux-gnu` com `--use-napi-cross`: o gcc embutido no [`@napi-rs/cross-toolchain`](https://github.com/napi-rs/cross-toolchain) é antigo demais para o `aws-lc-sys` ([cross-toolchain#4](https://github.com/napi-rs/cross-toolchain/issues/4)).

Duas soluções de contorno:

- Compile as partes em C com `clang` em vez do gcc embutido:

  ```sh
  TARGET_CC=clang TARGET_CXX=clang++ napi build --release --target aarch64-unknown-linux-gnu --use-napi-cross
  ```

- Use `--cross-compile` (`-x`) em vez de `--use-napi-cross`.

Veja a seção [Dependências nativas](../cross-build#depend%C3%AAncias-nativas) do guia de Compilação cruzada para os outros problemas recorrentes de compilação cruzada de C/C++.

## Compilar para `Windows i686`

Há um erro de `codegen` ao compilar para o destino `i686-windows-*`: [Rust issue 67497](https://github.com/rust-lang/rust/issues/67497).

Existe uma solução alternativa para evitar esse problema:

- Defina `lto` como falso, se você não tiver definido lto em seu `Cargo.toml`, o valor é falso por padrão, você pode ignorar este.
- Defina `codegen-units` para `32` (ou maior). O valor padrão de `codegen-units` é `16` por padrão ao compilar para release, você pode definir `CARGO_PROFILE_RELEASE_CODEGEN_UNITS=32` e `CARGO_PROFILE_RELEASE_LTO='false'` para deixar o compilador feliz ao direcionar `i686-windows-*`. Aqui está um [exemplo](https://github.com/napi-rs/package-template/blob/50ecdec7c7d31c60b693d5d52be6e13ba9b32bf8/.github/workflows/CI.yaml#L89-L91).
