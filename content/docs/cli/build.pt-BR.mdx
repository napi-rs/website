## Uso


## Flags de compilação cruzada

`napi build` tem três flags de compilação cruzada: `--use-napi-cross`, `--cross-compile` (`-x`) e `--use-cross`. As três são experimentais: o comportamento pode mudar entre releases minor.

As flags recomendadas são `--use-napi-cross` para targets Linux glibc em um host Linux x64/arm64, e `--cross-compile` (`-x`) para targets Windows MSVC a partir de um host não-Windows e para targets musl. `-x` também é o fallback para targets glibc, macOS e FreeBSD quando a configuração preferida não está disponível no seu host. Targets Android, WASI e OpenHarmony não precisam de nenhuma flag de cross: a CLI configura as toolchains deles a partir de variáveis de ambiente da plataforma. `--use-cross` é legada e não recomendada, e as builds baseadas em imagens Docker estão descontinuadas. Esta página é uma referência do que cada flag faz. Para escolher a flag certa para o seu host e target, veja [Compilação cruzada](../cross-build). Para especificidades de Alpine/musl, veja o [FAQ](../more/faq#compilar-para-linux-alpine).

Cada flag muda exatamente uma coisa na build:

| Flag                    | O que ela muda                                      | Comando resultante                                                               |
| ----------------------- | --------------------------------------------------- | -------------------------------------------------------------------------------- |
| _(nenhuma)_             | nada                                                | `cargo build --target <triple>`                                                  |
| `--use-cross`           | apenas o **binário**                                | `cross build --target <triple>`                                                  |
| `--cross-compile` / `-x` | apenas o **subcomando** (mais dois efeitos colaterais de env) | `cargo zigbuild --target <triple>` ou `cargo xwin build --target <triple>`       |
| `--use-napi-cross`      | apenas **variáveis de ambiente** (linker, CC, sysroot) | continua `cargo build --target <triple>`                                         |

### Escolha exatamente uma

<Callout type="warning">
  Essas flags não se combinam. Escolha exatamente uma. Duas das combinações
  imprimem um aviso dizendo que uma das flags será ignorada, mas os dois
  mecanismos na verdade continuam ativos.
</Callout>

| Combinação                             | O que realmente acontece                                                                                                                                          |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--use-cross` + `--cross-compile`      | Erro fatal: a CLI se recusa a compilar. (Ela ainda pode instalar automaticamente o cargo-xwin ou o cargo-zigbuild antes de dar o erro.)                           |
| `--use-napi-cross` + `--use-cross`     | Imprime um aviso dizendo que `--use-cross` será ignorada — mas o `cross` roda mesmo assim, com caminhos do host injetados no ambiente dele. Não combine.          |
| `--use-napi-cross` + `--cross-compile` | Imprime um aviso dizendo que `--cross-compile` será ignorada — mas o `cargo zigbuild` roda mesmo assim, com o env do napi-cross também definido (o linker do napi-cross vence o do zig). Não combine. |
| `--watch` + `--cross-compile`          | Produz um comando inválido (`cargo watch ... -- cargo build zigbuild`). Não combine.                                                                              |

### Pré-requisitos

| Flag                     | Instalado para você                                                                                                                                                                                | Você precisa providenciar                                                                                                                                                                                                                                 |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `-x`, target não-Windows (cargo-zigbuild) | <Green>cargo-zigbuild</Green> via `cargo install` no primeiro uso (o que pode ser lento). | `zig` no `PATH`. A CLI nunca instala nem verifica o zig; o cargo-zigbuild dá erro se ele estiver faltando.                                                                                                                                                 |
| `-x`, target Windows (cargo-xwin) | <Green>cargo-xwin</Green> via `cargo install`, no primeiro uso. Ele baixa por conta própria a CRT da Microsoft e o SDK do Windows (a licença da Microsoft se aplica). | `clang` (ex: `apt install clang` / `brew install llvm`) — o zig **não** é usado neste caminho. Para dependências que compilam assembly, também as ferramentas LLVM (`rustup component add llvm-tools`). A CLI não verifica nenhum desses.                  |
| `--use-cross`            | Nada.                                                                                                                                                                                              | O binário `cross` (um binário ausente falha com `spawn cross ENOENT`), mais um Docker >= 20.10 ou Podman >= 3.4 em execução.                                                                                                                              |
| `--use-napi-cross`       | A toolchain gcc, baixada automaticamente do npm (<Green>@napi-rs/cross-toolchain</Green>) e cacheada em `~/.napi-rs/cross-toolchain`.                                                              | `npm` no `PATH` e um host Linux x64 ou arm64. A CLI não verifica o host: no macOS ou no Windows a toolchain ainda é baixada sem erro, e o gcc de Linux então falha na hora do link. Qualquer falha aqui só imprime um aviso e a build continua como `cargo build` puro. |

### Exemplos

Um comando de copiar e colar por flag:

```sh
# Linux glibc targets, from a Linux x64/arm64 host
napi build --release --target aarch64-unknown-linux-gnu --use-napi-cross

# Windows MSVC from a macOS/Linux host, musl, or the zigbuild fallback cases
napi build --release --target x86_64-unknown-linux-musl --cross-compile

# Legacy container build (not recommended)
napi build --release --target x86_64-unknown-linux-gnu --use-cross
```

## O que o `napi build` executa

`napi build` é um wrapper em torno de um comando executado mais um conjunto de variáveis de ambiente. Esta seção detalha os dois.

### O comando

| Modo                                                              | Comando executado                                                            |
| ----------------------------------------------------------------- | --------------------------------------------------------------------------- |
| Sem flag de cross                                                 | `cargo build --target <triple>`                                              |
| `--use-napi-cross`                                                | `cargo build --target <triple>` (apenas o env muda)                          |
| `--use-cross`                                                     | `cross build --target <triple>` (mesmos argumentos, mesmo env calculado no host) |
| `--cross-compile`, plataforma do target é Windows, host não é Windows | `cargo xwin build --target <triple>` (`XWIN_ARCH=x86` é definido para `i686`) |
| `--cross-compile`, qualquer outro target                          | `cargo zigbuild --target <triple>`                                           |
| `--cross-compile`, target é Windows, host é Windows               | avisa e então roda um `cargo build --target <triple>` puro                   |

`--cross-compile` escolhe o cargo-xwin pela **plataforma** do target: todo triple `*-windows-*` compilado a partir de um host não-Windows é roteado pelo cargo-xwin — incluindo `x86_64-pc-windows-gnu`, não só MSVC. Ser roteado não é o mesmo que ser suportado, porém: o cargo-xwin lida apenas com MSVC, então para windows-gnu ele não configura nada e a build falha na hora do link com ``error: linker `x86_64-w64-mingw32-gcc` not found``. Compile esse target sem `-x`, com uma toolchain mingw-w64 — veja a nota sobre windows-gnu em [Receitas por target](../cross-build#receitas-por-target). Todo target não-Windows passa pelo cargo-zigbuild — a CLI não mantém nenhuma lista de targets suportados, e usa o zigbuild mesmo quando o target coincide com o host.

Se a variável de ambiente `CARGO` estiver definida, a CLI executa esse binário no lugar — em todos os modos, inclusive `--use-cross`, onde ele substitui silenciosamente o `cross`.

### RUSTFLAGS

- Qualquer target `*musl*`: a CLI adiciona `-C target-feature=-crt-static` ao `RUSTFLAGS`.
- `--strip`: a CLI adiciona `-C link-arg=-s`.

Ambos são aplicados através da variável de ambiente `RUSTFLAGS` exportada. O Cargo dá precedência à variável de ambiente sobre os `rustflags` do `.cargo/config.toml`, então, uma vez que a CLI a exporta, os rustflags do seu `.cargo/config.toml` são ignorados. Se você precisa de flags extras, adicione-as à variável de ambiente `RUSTFLAGS`, não ao `.cargo/config.toml`.

### Compilador C

Quando tanto `TARGET_CC` quanto `CC` estão definidas, `TARGET_CC` vence (desde `@napi-rs/cli` 3.0.0-alpha.92).

### Linkers padrão para targets menos comuns

Sem `--cross-compile`, estes targets recebem `CARGO_TARGET_<T>_LINKER` apontando para um gcc cruzado que **você mesmo precisa instalar**. A CLI define a variável de ambiente sem verificá-la: se o binário estiver faltando, a build falha na hora do link. A sua própria variável de ambiente `CARGO_TARGET_<T>_LINKER` sempre vence. Com `--cross-compile` esta tabela é pulada — o link é delegado ao zig ou ao xwin.

| Target                          | Linker definido pela CLI        |
| ------------------------------- | ------------------------------- |
| `aarch64-unknown-linux-musl`    | `aarch64-linux-musl-gcc`        |
| `loongarch64-unknown-linux-gnu` | `loongarch64-linux-gnu-gcc-13`  |
| `riscv64gc-unknown-linux-gnu`   | `riscv64-linux-gnu-gcc`         |
| `powerpc64le-unknown-linux-gnu` | `powerpc64le-linux-gnu-gcc`     |
| `s390x-unknown-linux-gnu`       | `s390x-linux-gnu-gcc`           |

### Android, WASI e OpenHarmony

Esses targets recebem o env de toolchain da CLI sempre que a plataforma do target corresponde — com ou sem qualquer flag de cross — mas cada plataforma tem suas próprias condições:

- **Android**: o env de linker/CC/AR é construído a partir de `ANDROID_NDK_LATEST_HOME`. Uma variável ausente só imprime um aviso — as variáveis de ambiente ainda são definidas, com caminhos quebrados, e a build falha na hora do link. Toda a configuração é pulada quando o próprio host é Android.
- **WASI**: `EMNAPI_LINK_DIR` é sempre definida para o emnapi embutido (a CLI dá erro se as versões de `emnapi`, `@emnapi/core` e `@emnapi/runtime` divergirem). O env de linker/CC do wasi-sdk só é definido quando `WASI_SDK_PATH` está definida **e** o diretório existe — caso contrário, o link recua para o padrão do cargo, o `rust-lld` que acompanha o rustup.
- **OpenHarmony**: o env é construído a partir de `$OHOS_SDK_PATH/native`, ou de `OHOS_SDK_NATIVE` quando `OHOS_SDK_PATH` não está definida. Se nenhuma das duas estiver definida, a CLI avisa e não define nada.

## Passando flags para o Cargo

Flags depois de `--` serão repassadas para o comando cargo build. Por exemplo:

```sh
napi build -- --locked
```

Isso passará a flag `--locked` para o `cargo build`, resultando em `cargo build --locked`.

## Observações para `--js-package-name`

Na seção de [aprofundamento](../introduction/getting-started#aprofundamento), recomendamos que você publique seu pacote sob um [`escopo npm`](https://docs.npmjs.com/creating-and-publishing-scoped-public-packages/). Mas se você estiver migrando um pacote existente que não está sob um [`escopo npm`](https://docs.npmjs.com/creating-and-publishing-scoped-public-packages/) ou simplesmente não deseja que seu pacote esteja sob um [`escopo npm`](https://docs.npmjs.com/creating-and-publishing-scoped-public-packages/) , você pode acionar a [_deteção de spam do npm_](https://stackoverflow.com/questions/48668389/npm-publish-failed-with-package-name-triggered-spam-detection/54135900#54135900) ao publicar os pacotes de plataforma nativa. Como `snappy-darwin-x64` `snappy-darwin-arm64` etc...

Nesse caso, você pode publicar seus pacotes de plataforma sob um [`escopo npm`](https://docs.npmjs.com/creating-and-publishing-scoped-public-packages/) para evitar a [_deteção de spam do npm_](https://stackoverflow.com/questions/48668389/npm-publish-failed-with-package-name-triggered-spam-detection/54135900#54135900). E seus usuários não precisam se preocupar com os pacotes nativos da plataforma em `optionalDependencies`. Como [`snappy`](https://github.com/Brooooooklyn/snappy/), os usuários só precisam instalá-lo via `yarn add snappy`. Mas os pacotes nativos da plataforma estão sob o escopo `@napi-rs`:

```json
{
  "name": "snappy",
  "version": "7.0.0",
  "optionalDependencies": {
    "@napi-rs/snappy-win32-x64-msvc": "7.0.0",
    "@napi-rs/snappy-darwin-x64": "7.0.0",
    "@napi-rs/snappy-linux-x64-gnu": "7.0.0",
    "@napi-rs/snappy-linux-x64-musl": "7.0.0",
    "@napi-rs/snappy-linux-arm64-gnu": "7.0.0",
    "@napi-rs/snappy-win32-ia32-msvc": "7.0.0",
    "@napi-rs/snappy-linux-arm-gnueabihf": "7.0.0",
    "@napi-rs/snappy-darwin-arm64": "7.0.0",
    "@napi-rs/snappy-android-arm64": "7.0.0",
    "@napi-rs/snappy-android-arm-eabi": "7.0.0",
    "@napi-rs/snappy-freebsd-x64": "7.0.0",
    "@napi-rs/snappy-linux-arm64-musl": "7.0.0",
    "@napi-rs/snappy-win32-arm64-msvc": "7.0.0"
  }
}
```

Para este caso, `@napi-rs/cli` fornece o `--js-package-name` para substituir a lógica de carregamento de pacotes gerados. Por exemplo, no `snappy`, temos um <Green>package.json</Green> assim:

```json
{
  "name": "snappy",
  "version": "7.0.0",
  "napi": {
    "name": "snappy"
  }
}
```

Sem a flag `--js-package-name`, `@napi-rs/cli` vai gerar a binding JavaScript para carregar pacotes nativos da plataforma para você:

```js {10,22} filename="index.js"
switch (platform) {
  case 'darwin':
    switch (arch) {
      case 'x64':
        localFileExisted = existsSync(join(__dirname, 'snappy.darwin-x64.node'))
        try {
          if (localFileExisted) {
            nativeBinding = require('./snappy.darwin-x64.node')
          } else {
            nativeBinding = require('snappy-darwin-x64')
          }
        } catch (e) {
          loadError = e
        }
        break
      case 'arm64':
        localFileExisted = existsSync(join(__dirname, 'snappy.darwin-arm64.node'))
        try {
          if (localFileExisted) {
            nativeBinding = require('./snappy.darwin-arm64.node')
          } else {
            nativeBinding = require('snappy-darwin-arm64')
          }
        } catch (e) {
          loadError = e
        }
        break
      default:
        throw new Error(`Unsupported architecture on macOS: ${arch}`)
    }
    break
    ...
}
```

Isso não é o que queremos. Então, construa com `--js-package-name` para substituir o `package name` no arquivo de binding JavaScript gerado: `napi build --release --platform --js-package-name @napi-rs/snappy`. Em seguida, o arquivo JavaScript gerado se tornará:

```js {10,22} filename="index.js"

```
