# Support-matrix badge — design

**Status:** approved design, pre-implementation
**Date:** 2026-07-10
**Related precedent:** `docs/superpowers/plans/2026-07-06-sponsors-image-router.md` (the sponsors image router — the direct template for the edge-render mechanics)

## Problem

Every napi-rs ecosystem package that wants to show its platform/runtime support in
its README hand-maintains a set of SVGs. `@napi-rs/lzma` is the reference case: it
ships six files — `assets/support-{node,platforms,browser}-{light,dark}.svg` —
embedded via `<picture>` (light/dark) plus a collapsible text table. These are
authored and regenerated by hand, drift from the package's real `napi.targets` /
CI, and every package reinvents them.

## Goal

Host a public, parameterized image on `napi.rs` that renders a support matrix from
query parameters, so any ecosystem package embeds one URL instead of maintaining
its own SVGs. The URL is **self-contained** (all facts in the query) so it renders
without any server-side lookup, caches at the edge per query string, and works
before a package is published.

The rendered image reproduces the existing napi-rs support-matrix visual: three
stacked cards — **Node.js**, **Platforms** (native targets grouped by OS), and
**Browser** (`wasm32-wasi`) — with a three-tier status legend (CI-tested green /
non-blocking amber / built-untested gray).

## Scope

**In scope (v1):**

1. The rendering service — `routes/support-matrix.svg.ts` + `routes/support-matrix.png.ts`, a triple→display data model, and a satori renderer.
2. A docs page documenting the query contract + copy-paste embed snippets (en/cn/pt-BR).
3. An interactive URL-builder page on `napi.rs`.

**Out of scope (deferred):**

- Dogfooding: replacing the hardcoded landing `components/landing/support-matrix.tsx` with this service. (The landing component is a different, simpler 7×5 grid; migrating it is a follow-up.)
- A `@napi-rs/cli` subcommand that emits the URL (lives in the `napi-rs` repo, not this one).
- Localizing the builder page (en-only for v1).

## Decisions (locked with the user)

| #   | Decision          | Choice                                                                                                   |
| --- | ----------------- | -------------------------------------------------------------------------------------------------------- |
| 1   | Input model       | **Query-encoded, self-contained** — no `?pkg=` lookup, no network fetch                                  |
| 2   | Deliverable       | **One combined image** (Node + Platforms + Browser stacked), **SVG + PNG**                               |
| 3   | Status encoding   | **Per-tier lists** — `tested=` / `nonblocking=` / `untested=`                                            |
| 4   | Convenience       | **`full` keyword** + **`omit`** (subtract triples or OS groups)                                          |
| 5   | Node card         | **Derived** from `engines` semver + `nodeTested` (not a dumb raw string)                                 |
| 6   | Target vocabulary | **Full Rust triples** in the query (copy-paste from `napi.targets`), rendered as short chips ("x64 gnu") |

## Endpoints

```
GET /support-matrix.svg?<query>   → image/svg+xml; charset=utf-8   (GitHub, <picture> theme-switch)
GET /support-matrix.png?<query>   → image/png                       (npm, crates.io — no <picture>)
```

- **Top-level paths** (not under `/docs`, `/blog`, `/changelog`), so they are not caught by the `void.json` i18n rewrites. Mirrors `/sponsors.svg|png`.
- **Must not** set `Cross-Origin-Resource-Policy` / `Cross-Origin-Embedder-Policy` (those would block cross-origin `<img>` embeds). Top-level path + no CORP header satisfies this, exactly as the sponsors routes do.
- **Caching:** `Cache-Control: public, s-maxage=86400, max-age=86400, stale-while-revalidate=604800`. The image is a pure function of the query; Cloudflare's default cache key includes the full query string, so every distinct query caches independently at the edge. **No KV/R2 warm cache and no cron** (unlike sponsors, there is no external data to refresh).
- **Malformed input never 500s** — unknown/invalid triples and params are skipped, following the `parseTheme` "normalize untrusted query param" pattern in `lib/sponsors-image/theme.ts`.

## URL contract

| Param         | Meaning                                                                             | Example                                                 |
| ------------- | ----------------------------------------------------------------------------------- | ------------------------------------------------------- |
| `tested`      | Rust triples (or `full`) that are CI-tested — green                                 | `full` or `x86_64-unknown-linux-gnu,…`                  |
| `nonblocking` | Triples tested non-blocking (`continue-on-error`) — amber                           | `powerpc64le-unknown-linux-gnu,s390x-unknown-linux-gnu` |
| `untested`    | Triples built but not exercised — gray                                              | `riscv64gc-unknown-linux-gnu,aarch64-linux-android`     |
| `omit`        | Triples **or** OS-group names to remove entirely                                    | `android,freebsd` or `aarch64-unknown-freebsd`          |
| `engines`     | Node semver range → the "vMIN → vMAX" headline + excluded gaps                      | `^22.20 \|\| ^24.12 \|\| >=25`                          |
| `nodeTested`  | Node majors that are CI-green (rest of range = "supported, not in CI")              | `22,24`                                                 |
| `wasm`        | `1` forces the Browser section (also auto-on if a `wasm32-wasi*` triple is present) | `1`                                                     |
| `name`        | Cosmetic title / alt-text                                                           | `@napi-rs/lzma`                                         |
| `theme`       | `light` (default) or `dark`                                                         | `dark`                                                  |

### Resolution order (query → matrix model)

```
1. seed tiers: expand `full` wherever it appears        (tested=full → every scaffold triple seeded "tested")
2. explicit triples override the seed                    (a triple in untested= downgrades it; a triple not
                                                          in `full` is simply added to its tier)
3. omit removes targets entirely                         (by triple, or by OS-group name)
4. group survivors by OS → the Platforms section
5. Node section  ← engines + nodeTested + NODE_LATEST
6. Browser section ← present iff a wasm32-wasi* triple survived, or wasm=1
```

Precedence rule: **an explicit tier listing always wins over a `full` seed.** If the
same triple appears in two explicit tiers, the most severe (untested > nonblocking >
tested) wins (deterministic; documented).

### Worked example — lzma via `full`

```
/support-matrix.svg?theme=dark
  &tested=full
  &nonblocking=powerpc64le-unknown-linux-gnu,s390x-unknown-linux-gnu
  &untested=riscv64gc-unknown-linux-gnu,aarch64-linux-android,
            arm-linux-androideabi,wasm32-wasi-preview1-threads
  &engines=^22.20%20||%20^24.12%20||%20>=25
  &nodeTested=22,24
```

Produces lzma's exact matrix (11 CI-tested, 2 non-blocking, 4 built-untested, wasm
browser) in ~4 short params instead of 17 triples.

## Data model — `lib/support-matrix/targets.ts` (pure, unit-tested, no deps)

```ts
type OsGroup =
  | 'Linux'
  | 'Windows'
  | 'macOS'
  | 'Android'
  | 'FreeBSD'
  | 'OpenHarmony'
  | 'Browser'
interface TargetInfo {
  os: OsGroup
  arch: string
  abi?: 'gnu' | 'musl'
  label: string
}

const TARGETS: Record<Triple, TargetInfo> // the full napi target universe (~20 entries)
const FULL: Triple[] // what `full` expands to — the napi-new scaffold set
const OS_GROUPS: Record<string, Triple[]> // for omit=android|freebsd|openharmony|windows|macos|linux
const ALIASES: Record<string, Triple> // normalize to one canonical triple
```

**Alias normalization** (both spellings accepted, resolve to one entry):

- `wasm32-wasi-preview1-threads` ⇄ `wasm32-wasip1-threads`
- `arm-linux-androideabi` ⇄ `armv7-linux-androideabi`

**`FULL`** = the scaffold set the landing "16 native targets" reflects (macOS x64/arm64;
Windows MSVC x64/x86/arm64; Linux gnu x64/arm64/armv7; Linux musl x64/arm64; Android
arm64/armv7; FreeBSD x64; WASI threads). Targets outside this set (ppc64le, s390x,
riscv64, openharmony, freebsd-arm64, armv7-musl, riscv64-musl, darwin-universal) exist
in `TARGETS` and can be added to a tier explicitly, but are not seeded by `full`.

**Display label** derives from `{arch, abi}` — e.g. `x86_64-unknown-linux-gnu` → `x64 gnu`,
`aarch64-unknown-linux-musl` → `arm64 musl`, `i686-pc-windows-msvc` → `x32`,
`powerpc64le-unknown-linux-gnu` → `ppc64le`. Chips group under their `os`.

## Node section — `lib/support-matrix/node.ts` (pure, unit-tested)

Input: `engines` semver range, `nodeTested` majors, a `NODE_LATEST` constant.
Output: `{ headline: 'v22.20 → v26', excluded: 'Node 23 and 24.0–24.11 are excluded',
pills: [{major, floor, tested}] }`.

Derivation: parse the range, take the lowest satisfied version as the floor, the
`NODE_LATEST` constant (bumped as Node releases) as the ceiling, compute the excluded
gaps between satisfied majors, and mark majors present in `nodeTested` green (rest of
the range gray "supported, not in CI"). Uses a minimal in-repo semver-range check to
avoid an edge dependency; if the ranges seen prove broader than the minimal parser
handles, fall back to the `semver` package (small, worker-compatible).

## Renderer — `lib/support-matrix/{card,render,theme}.ts`

Reuses the sponsors edge-image mechanics **verbatim**, importing the already-generic
helpers rather than refactoring them:

- `satori/standalone` + `init(yogaWasm)` — the `ensureYoga` memoized-guard pattern.
- `@resvg/resvg-wasm` for PNG — **reuse `lib/sponsors-image/resvg.ts`** (`ensureResvg`, `svgToPng`).
- Fonts: **reuse `lib/sponsors-image/fonts.ts`** (`loadFonts`, `readAsset`) — Manrope 700/500 from `public/fonts/` via the `ASSETS` binding and `c.req.url` as base.
- `.wasm` imported at the top of the route files only (turns into `WebAssembly.Module` at the edge); typed by the existing `lib/sponsors-image/wasm.d.ts` ambient (`declare module '*.wasm'`). Route passes the module into the lib units so they stay Node-testable.

**Visual:** three stacked cards in one rounded container, reproducing the reference
screenshot / lzma design — card header with check icon + title + subtitle, OS-grouped
chip rows, per-tier legend line. Tier colors: green (tested), amber (non-blocking),
gray (built-untested). Light + dark palettes in `theme.ts`. Icons are inline SVG paths
(satori has no icon font). Fixed width ~900px; height auto from content. satori embeds
fonts as vector paths, so the SVG is self-contained and Camo-safe (no external refs),
and resvg needs no fonts.

## URL-builder page — `pages/support-matrix.island.tsx` → `/support-matrix`

Interactive island (en-only, v1). Distinct path from the `.svg`/`.png` routes — no clash.

```
paste napi.targets  →  per-target tier toggles (default tested)  |  [✓ full scaffold set]
engines input  +  tested-node-majors input  +  theme light/dark
live preview: <img src="/support-matrix.svg?…current-params"> (dogfoods the endpoint — WYSIWYG)
outputs: copy <picture> snippet  |  copy PNG-markdown snippet
```

UI lives in `components/support-matrix-builder/*`.

## Docs — `content/docs/more/support-matrix-badge.{en,cn,pt-BR}.mdx`

Query-param reference, `full`/`omit` + precedence rules, copy-paste `<picture>` and PNG
markdown snippets, the worked lzma example, and a link to the builder. Authored in
`content/` (source) → regenerated `pages/` mirror; three locales per repo convention.

## File layout

```
routes/
  support-matrix.svg.ts        thin adapter: parse query → render SVG   (mirror routes/sponsors.svg.ts)
  support-matrix.png.ts        thin adapter: parse query → render PNG
lib/support-matrix/
  targets.ts                   TARGETS + FULL + OS_GROUPS + ALIASES     (pure, tested)
  resolve.ts                   query → { node, platforms[], browser }   (pure, tested — full/omit/precedence)
  node.ts                      engines + nodeTested → node card model   (pure, tested — range/gaps)
  card.ts                      model → satori JSX → SVG string
  render.ts                    orchestrator → { body, contentType } for svg|png
  theme.ts                     light/dark palettes + tier colors
reuse (no change): lib/sponsors-image/{resvg,fonts}.ts, lib/sponsors-image/wasm.d.ts
components/support-matrix-builder/*    builder UI
pages/support-matrix.island.tsx        builder page → /support-matrix
content/docs/more/support-matrix-badge.{en,cn,pt-BR}.mdx   docs
```

## Testing

- **Pure units (vitest, no edge):** `targets` (alias normalization, label derivation, OS grouping), `resolve` (`full` seeding, `omit` by triple + OS group, tier precedence, malformed-input skipping), `node` (floor/ceiling, excluded-gap computation, tested vs supported pills).
- **Renderer:** string-snapshot the generated SVG for three fixtures — lzma, `tested=full`, and a minimal single-target case.
- **Routes:** assert `Content-Type` for `.svg`/`.png` and that malformed queries return a valid image (never 500).
- **Builder:** light smoke test that params round-trip into the preview `src` and snippet output.

## Constraints & notes

- Register routes with `void prepare` (adds them to `.void/routes.d.ts`). `KV`/`STORAGE`/`ASSETS` bindings already enabled — this feature only needs `ASSETS` (fonts).
- The middleware pipeline (`01.head` / `02.i18n-fallback` / `03.page-path`) does not touch image routes: head injection is gated to `text/html`, i18n fallback only matches known page paths under `/cn|/pt-BR`, and the route path is top-level.
- Keep `.wasm` imports in the route files only; pass modules into `lib/` units to preserve Node unit-testability.

## Open items (resolved as "no")

- No warm cache / cron — the param space is open and the render is a pure function; edge `Cache-Control` is sufficient (the sponsors plan itself calls an in-isolate memo "YAGNI" given `s-maxage`).
- No edge `semver` dependency unless a minimal range parser proves insufficient during implementation.
