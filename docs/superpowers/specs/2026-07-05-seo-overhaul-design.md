# SEO Overhaul — Design

Date: 2026-07-05
Status: Draft (awaiting review)
Scope: source-level SEO for the Void/Cloudflare napi.rs build in this repo.
Out of scope: deployment/hosting, the Vercel→Void cutover, and any 301 redirect
map for old URL forms (per explicit direction — "no need to care where the
project is deployed").

---

## 1. Context

The site runs on **Void 0.10.x** (Vite + React 19) on Cloudflare Workers.
`dist/ssr/wrangler.json` sets `run_worker_first: ["/**"]`, so the worker — and
all three middlewares — runs on **every** request, prerendered pages included.

`<head>` is produced in three layers (`config < middleware < page`):

```
void.json head           → site-wide defaults (title template, static og/twitter, favicons)
*.md frontmatter /        → per-page title + description (content)
  defineHead sidecars
middleware/01.head.ts     → post-next() body rewrite: injects og:title/description/url,
                            per-locale <html lang>, markdown-alternate link
```

Void has **no post-render head hook** and `headDefaults` resolves _before_ the
page renders, which is exactly why per-page og tags are injected by rewriting
the finished HTML after `next()`. Any new head signal that is **derivable from
path + locale** belongs in that same rewrite; anything that is **content**
(descriptions, article dates) belongs at the source.

Rendering is healthy: docs/blog prerender to real static HTML, so crawlers get
full content, not a JS shell.

## 3. Architecture decision

**Chosen: extend `middleware/01.head.ts` as the single injection point for
derivable head signals, fed by a build-time-baked route→locales map.**

```
                       imports (edge-safe, no fs)
  route-map.gen.ts  ─────────────────────────────►  middleware/01.head.ts
  { path → locales[], lastmod }                         │  injects per request/prerender:
        ▲                                                ├─ <link rel=canonical>  (real served URL)
        │ generated by                                   ├─ <link rel=alternate hreflang> ×N + x-default
        │                                                ├─ fixed og:url (= canonical)
  scripts/generate-sitemap.mjs ──────────────────►  └─ JSON-LD (WebSite/Org/TechArticle/Breadcrumb/BlogPosting)
  (already walks pages/, has nav + hasMarkdownSource)     │
        │ also consumes the same map for                  └─ real <meta name=description> fallback
        ▼ <lastmod> + <xhtml:link> alternates

  content frontmatter / defineHead → per-page description, blog date  (content, stays at source)
```

Why baked module: the middleware runs at the edge and cannot walk the
filesystem to learn which locales serve a given page. A build-generated module
is the established pattern in this repo (`changelog-data.gen.ts`). The sitemap
generator already computes every route and already knows per-locale existence
via `hasMarkdownSource()` — so it is the natural producer of the map, and the
map keeps sitemap + middleware in lockstep from one source.

Rejected alternatives:

- **Per-page `defineHead` for everything** — docs pages are generated `.md`;
  every new tag would force `convert-content.mjs` changes and per-file upkeep.
  Doesn't scale.
- **A Void head plugin/hook** — none exists post-render; that absence is the
  reason the current og injection is a body rewrite.

## 4. Phases

Each phase is independently shippable and ordered by impact ÷ effort.

### Phase 1 — Canonical + i18n signals _(high impact, low risk)_

Add to the `01.head.ts` post-`next()` rewrite:

- **`<link rel="canonical">`** = the **real served URL**: `https://napi.rs` +
  cleaned `publicPath` (unprefixed en, `/cn/…` and `/pt-BR/…` kept, no trailing
  slash; root = `https://napi.rs/`). This is a _new_ helper `selfCanonical()`,
  distinct from today's always-`/en/`-prefixed `canonicalUrl()`.
- **Fix `og:url`** to equal the canonical (drop the `/en/` mismatch).
- **i18n-fallback dedupe:** a `/cn/…` or `/pt-BR/…` request with no translation
  is served en content (`x-i18n-fallback: 1`, set by `02.i18n-fallback.ts`,
  readable on `res` after `next()`). For those, canonical points to the **en**
  URL so duplicate content consolidates.
- **`<link rel="alternate" hreflang="…">`** for every locale that has a _real_
  translated source (`hasMarkdownSource`), plus `hreflang="x-default"` → en.
  Locales that only exist via fallback get **no** hreflang entry (they aren't a
  distinct translation).
- New **`lib/i18n/route-map.gen.ts`** (baked): `path (locale-neutral) → { locales, lastmod }`.

Acceptance: every page emits exactly one self-referencing canonical; multi-locale
pages emit reciprocal hreflang + x-default; fallback pages canonicalize to en;
`og:url === canonical`. Unit test the map + the URL helpers.

### Phase 2 — Meta descriptions _(high impact, low effort)_

- **Emit a real `<meta name="description">`** in the rewrite when the page
  declared none (today only `og:description` gets the fallback). Prefer a
  meaningful per-page value; fall back to `DEFAULT_DESCRIPTION` only as last
  resort.
- **Author the ~22 missing descriptions** at the source: blog posts (all),
  concept docs (`class`, `enum`, `object`, `exports`, `naming-conventions`,
  `module-init`) across locales, and changelog sidecars. Descriptions live in
  `content/**` frontmatter (flows to `pages/**/*.md` via `convert-content.mjs`)
  or in the `*.server.ts` `defineHead` for island pages.

Acceptance: no indexable page ships without a `<meta name="description">`; audit
grep returns zero description-less routes.

### Phase 3 — Structured data (JSON-LD) _(medium impact)_

Injected in the rewrite, `<script type="application/ld+json">`, chosen by route
shape:

- **Home** (`/`, `/cn`, `/pt-BR`): `WebSite` + `Organization` (name, url, logo,
  sameAs → GitHub/Twitter/Discord).
- **Docs** (`/docs/**`): `TechArticle` (headline = title, inLanguage) +
  `BreadcrumbList` derived from the path + `lib/nav`.
- **Blog** (`/blog/**`): `BlogPosting` (headline, author, inLanguage,
  datePublished). `datePublished` depends on blog `date` frontmatter (Phase 5b);
  either pull 5b forward, or fall back to the git commit time already fetched in
  Phase 4 until real dates land.

Acceptance: Google Rich Results test passes for one home, one docs, one blog URL.

### Phase 4 — Sitemap enrichment _(medium, low effort)_

Extend `scripts/generate-sitemap.mjs` (consuming the Phase 1 route map):

- `<lastmod>` per URL from **git commit time** of the underlying content file
  (spawn `git log -1 --format=%cI -- <file>` at build; fall back to build time
  if git history is unavailable, e.g. shallow CI checkout).
- `<xhtml:link rel="alternate" hreflang="…">` alternates per URL (same rule as
  Phase 1: real translations + x-default).

Acceptance: sitemap validates; every multi-locale URL carries reciprocal
`xhtml:link`s; each URL carries a plausible `lastmod`. Keep the generator
deterministic/idempotent (byte-identical re-runs given identical git state).

### Phase 5 — Per-page OG images + blog dates _(medium impact, higher effort)_

**5a — Dynamic OG images.** Build-time generate a branded PNG per docs/blog page
(page title rendered on a template) using **Satori → resvg** (`satori` +
`@resvg/resvg-js`; no headless browser). Emit into `dist/client/og/<route>.png`
as a new Vite build plugin alongside `sitemapPlugin()`. Wire per-page
`og:image` + `twitter:image` (+ `:width`/`:height`/`:alt`) in the rewrite,
falling back to the static `og-v2.png` for pages without a generated image
(home, landing).

**5b — Blog `date` frontmatter.** Add `date:` to every blog post's frontmatter
(currently the date lives in body prose, so posts are skipped from RSS). One
change feeds RSS `pubDate`, JSON-LD `datePublished`, and sitemap `lastmod`
accuracy.

Acceptance: each docs/blog page has a unique OG image resolving at its URL;
Twitter/OG validators show the per-page card; RSS now includes previously
skipped posts.

### Phase 6 — Head polish _(low impact, low effort)_

- cn / pt-BR landing: render a real `<h1>` (replace the `<p>` title component,
  or wrap it) so every locale home has exactly one h1.
- Link `apple-touch-icon`; add a minimal PWA `manifest.webmanifest`
  (name/short_name/icons/theme_color/background_color) + `<meta name="theme-color">`.
  Assets already exist in `public/img/`.
- Add `twitter:title` / `twitter:description` (mirror og), and
  `og:image:width`/`height`/`alt`.
- Add `alt` to the two bare `<img>` (`optimized-performance.tsx:146`,
  `link-preview.tsx:67`).

Acceptance: Lighthouse SEO + PWA installability checks pass; one h1 per home.

## 5. Testing

- **Unit:** route→locales map; `selfCanonical()` / hreflang helpers; sitemap
  `lastmod` + `xhtml:link` rendering (fixture git state).
- **Middleware:** given a rendered HTML fixture, assert exactly one canonical,
  correct hreflang set, JSON-LD validity, description fallback, idempotency
  (second pass injects nothing).
- **Build integration:** run the sitemap + OG plugins on a temp `dist/client`;
  assert deterministic bytes.
- **External validators (manual gate before merge):** Google Rich Results,
  Twitter/OG card, and an XML sitemap validator on a `void dev` build.

---

## 6. Risks / open questions

- **hreflang for fallback locales:** we deliberately omit hreflang for
  fallback-only locales and canonicalize them to en. Confirm this matches
  intent (alternative: emit hreflang pointing at the fallback URL — not
  recommended; it advertises duplicate content as a distinct translation).
- **OG image build cost:** Satori/resvg over ~90 pages adds build time; keep it
  incremental/cached if the delta is noticeable. Deferrable within Phase 5 if it
  regresses `void deploy` too much.
- **git lastmod in CI:** needs non-shallow history (`fetch-depth: 0`) or a
  graceful build-time fallback. Note in the deploy workflow.
- **Content authoring (Phase 2 descriptions, 5b dates):** these are copy tasks,
  not code — will need either the maintainer's words or a first draft to review.
